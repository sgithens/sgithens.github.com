I translated the calls in the pixmap and freetype font related areas to use cffi rather than straight ffi
yesterday, and it was rather straightforward with the exception of a few areas. The main issue is that it
wasn't enough just to change the foreign-alloc method and use the same fli dereferencing operations.
This is because under the covers cffi:foreign-alloc always creates a BYTE pointer and then does the
math to translate the size from your :type and :count parameters to the method. So even if you specified:

(cffi:foreign-alloc :unsigned-int :initial-element 0 :count 20)

it would still come out as BYTE. This caused a problem when I was still using it in conjunction with the
fli deference operation which assumes the type based off of the pointer. The cffi:mem-aref operation
always takes an explicit type, so switching to this fixed things with the memory access and the images
looked normal again.

As a bit of background, "fli" is the foreign language interface that is specific to lispworks for
interoperability with C libraries and functions. "cffi" is a common lisp library for more generalized foreign
function access across common lisp distributions, and to be portable it tends to use the foreign function
functionality specific to each common lisp implementation, which is slightly different for each one.
Because it uses the lispworks fli to implement the functionality when used from lispworks you can sort of
occasionally mix and max the methods from each library, except when you run in to issues and
assumptions like the above. But at the very least it let's me slowly migrate things over to generalized
portability without having to do it all at once.
